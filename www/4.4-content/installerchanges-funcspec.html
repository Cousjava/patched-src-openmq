4.0 Description:
4.1  MQ 4.4 Installers
In MQ 4.3, the OpenMQ installed a zip-based format and SJSMQ installed native pkging (Solaris/Linux) or zip-based (Windows, AIX).  In MQ 4.4, as we move to creating IPS packages for our product, both the OpenMQ and SJSMQ installers will install IPS "user images".  For those platforms where Update Center is not supported (AIX, other UNIX), we will still install an IPS user image but updates will not be available through the Update Center.

The difference between each of the installers per platform is the presence of the MQ C-API IPS Package, MQ Native IPS Package, and the NSS/NSPR IPS packages pre-installed within each of the IPS user images.   We chose to "preinstall" the native code within each of our user images rather than bootstrap the native code after the install to avoid risk of those who do not have a network connection.  AIX is not supported in Update Center so for that platform we cannot preinstall the NSS IPS packages.  Instead, we will have to include the libs within the MQ C-API IPS Package.

4.2  IPS
The MQ 4.4 IPS packages conform to the file-layout described in the Multi-Install File Layout Specification.  See the MQ IPS Packaging Functional Spec for a detailed file-layout.  In addition, while the MQ 4.3 zip-based installers did not install IPS user images, they did install in a very similar file-layout as the MQ 4.4 IPS packages.  Therefore, a zip-based install with MQ 4.3 will look very similar (if not identical) to what you will see in an MQ 4.4 install.  The MQ 4.4 install might have the extra IPS tools installed in the user image when compared with the MQ 4.3 install.

4.3 IPS Tools in Installed User Image (for supported platforms only)
We have a couple of options for a user to update their installed image.  If we have extra time, we would do #3.  If not, then we'll do #1 or #2.
  1. We could pre-install the platform's IPS tools within the user image.  The user could later run these tools to update their installed image.
  2. We could provide the skeletal pkg or updatetool versions in our IPS user image being installed.  The user could later run those tools to bootstrap the tools to their system.  Once the IPS tools are installed, the user would run them to update their installed image.
  3. We could provide an option in the installer to ask if they wish to install the Update Center tools upon installation.   The installer could popup the updatetool with a list of latest packages to install.
4.3  No Native Packaging Installers
In MQ 4.4, we will no longer provide an installer to install SVR4 packages on Solaris or install RPMs on Linux.  We will continue to create the native packaging in our builds but they will only be available for supported customers.

4.4  Remove Text Support in Installers
Text support in installers really never worked well (hangs, exceptions, crashes).  There have been many bugs filed against the MQ installers which as passed onto OpenInstaller.  Nobody is fixing bugs in OpenInstaller so we are dropping the text support feature for MQ 4.4.  Text support was previously supported only on Solaris/Linux.

4.4  OpenMQ vs. SJSMQ Installers
  • As in MQ 4.3, the MQ 4.4 OpenMQ installer will register a different swordfish ID as the SJSMQ installer.  
  • An OpenMQ install should show "Open Message Queue" product version strings while an SJSMQ install should show "Sun Java System Message Queue" strings.  This is differentiated with the mq-commerial IPS Package install.
4.4  No JRE/JDK Installed
A user will be required to have a JRE installed to run the installers.  In addition, no JRE/JDK will be installed with any of the installers.
4.5  OpenInstaller engine-wrapper script
(Probably can't do this for MQ 4.4 timeframe now)

The OpenInstaller framework originally provided an engine-wrapper script for products to call the main OpenInstaller engine class upon start up.  This engine-wrapper script is a shell script on Unix and a .vbs script on Windows.  If time permits, we'd like to bypass the engine-wrapper script and find a means to call the OI Engine class directly from our own Java class.
  1. We have multiple bugs filed against our installer for not i18n'ing any text in the engine-wrapper class (usage statements, error messages).  Using a Java class as a launcher will enable us to i18n the text in engine-wrapper via resource bundles.
  2. VBS on windows is very difficult to debug.  We've had many complaints from customers as to why we chose to use a .vbs script for our installer/uninstaller launcher.  We'd like to rid of engine-wrapper.vbs and implement our own Java class that contains the same functionality of engine-wrapper.vbs.
  3. VBS on windows Vista will not allow a user to double click the installer.vbs file and execute for security reasons.   We presume it is because .vbs files are notorious for spreading viruses.
  4. <more reasons here> 
4.6  Remove L10n Screen

The preinstalled IPS user image installed by the installer will already contain the localizations so we will need to remove the installer screen that asks if localizations need to be installed.
4.7  Swordfish IDs
There will be two different Swordfish IDs registered by the installers; one by the OpenMQ installer and one by the SJSMQ installer.  This should be distinguished when loggin into the sysnet registration page.

4.8  NSS Dependency

The mq-capi IPS package has a hard dependency on the nss-libs IPS pkg.  The former NSPR libs are now combined within the nss-libs IPS pkg.  This  nss-libs package will also be installed with the MQ Installers (except for AIX) inside the user image.  Because NSS is installed as a separate shared component, the location of nss is now at the same level as the "mq" directory (e.g., nss/lib/*).  On Solaris and Linux we added the library runpath inside the MQ-C-API library to find ../../nss/lib/*.so during runtime.  On windows, users will have to add ../../nss/lib to their %PATH% to locate the NSS libs.

15.0 Use Cases:

MQ Install (pkg needs to be installed)

  1. User downloads any MQ installer.
  2. MQ Installer is run to register and install MQ software for the platform being used.
  3. User runs pkg and gets prompted if they wish to install Update Center.
  4. User runs pkg and updates the install to the latest version.
MQ Install (pkg tools are an option in Installer)
  1. User downloads any MQ installer.
  2. Installer asks if he wishes to install Update Center.
  3. User selects "install Update Center".
  4. pkg tools are downloaded after the MQ install
  5. pkg or Update Tool is run to update the current MQ install.
