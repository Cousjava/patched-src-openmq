Synopsis:
Changes to the installers for OpenMQ 4.4 and SJSMQ 4.4.
Quick Questions:
  • Difficulity: Medium

 

Description:
In MQ 4.3, there was a set of file-based installers that installed zip files for all platforms for OpenMQ and there was a corresponding set of native installers for the SJSMQ product.

For MQ 4.4, we will be making some changes to the installers. Here is a summary of the changes to the installers:

  • The OpenMQ installers will install IPS pkgs rather than zip files. Except for the location of the shared components, the final bits layed out onto the filesystem should be similar to what you see in MQ 4.3.
  • The SJSMQ solaris and linux installers installed native pkgs for MQ 4.3, but for MQ 4.4, they will now install IPS pkgs.
  • The SJSMQ windows and AIX installers installed zip files in MQ 4.3, but will now install IPS. Again, except for the location of the shared components, the final bits layed out onto the filesystem should be similar to what you see in MQ 4.3.
  • We will continue to build Solaris pkgs and RedHat RPMs but those bits will not be available in a GUI installer. Native pkgs will still be available only for supported customers who are not ready to migrate to IPS. These pkgs will be posted to usr.sfbay, a support server, and possible a JES server for former JES customers.
  • Text support in the installer will be dropped in MQ 4.4. The text mode libraries will still be in the installer image but the "-t" text option will no longer be public. If text mode works, then great, otherwise is it no longer supported.
  • If we have time, we'd like to get rid of the OI engine-wrapper script and call the OI EngineBootstrap class directly to fix i18n bugs and get rid of .vbs.
  • Still under investigation is how many installers we will need. At most, we will need:
    • OpenMQ Sparc, X86, Linux, AIX, Windows Installer (IPS)
    • SJSMQ Sparc, X86, Linux, AIX, Windows Installer (IPS)
    • OpenMQ Unix Installer (IPS)
    • The installers will not install a JDK/JRE. A user is required to have a JRE on the system in order for the installer to run.
    • For the installers that install the C-API pkg, it should also install an nss-libs and nss-utils IPS pkgs.
      • an IPS pkg available.

    If we can build the Unix installers to bootstrap the correct C-API pkg and the Server Native package, then we might be able to get away with a Unix-only installer. Otherwise, we'll build a separate installer for sparc, x86, linux, AIX for OpenMQ and SJSMQ.

Requestor:
With the move to the IPS pkging format, our installers will need to migrate over to installing IPS instead of zips. In addition, as IPS becomes more prevalent, native install formats will gradually be phased out and we will move our native installers over to IPS installers.
Non-feature:
This one-pager does not describe the IPS breakdown and exact file layout. See the IPS Packaging one-pager and IPS spec for more info on the IPS breakdown. 

This also does not describe our deliverables that are not installer-based (e.g., deliverables to GFv3, JES, OpenSolaris).
Dependencies:
  • OpenInstaller
  • IPS Packaging
Competitive Analysis:

Phases:
  • Investigate how to bootstrap the C-API and Server Native pkg from our installer. What if they are on MacOS where there is no C-API?
  • Changes to IPS files.
  • Create OpenMQ installers for each platform to install IPS.
  • Create SJSMQ installers for each platform to install IPS.
Direction of Implementation:
We will continue to use the OpenInstaller framework to install the product initially. Any updates to the product will probably be done through the Update Framework.
 
 
Interfaces:
  • New installer bundle names.
  • Default locations for installers.
Test Impact:
Testing is very similar to MQ 4.3 installer testing. We will have OpenMQ installers for each platform and SJSMQ installers for each platform. For any installers that used to install MQ 4.3 zips, the final layout should look similar except for the shared component (nss) locations.  On platforms where Update Center is supported, testing should be done to update the product (if updates posted) to the latest bits.
Doc Impact:
Install Guide will need to be updated. The installer screens probably won't change but we will have to change install locations, SVR4 pkgs and rpms -> IPS pkgs, shared components installed, how to get updates. We will also need to document the note under Issues (see below).

Issues:
OI writes out install data (product version, oi versions) to the system. If a user updates their product through the Update Center, the product version in OI and the product version actually installed on the system will get out of sync.
